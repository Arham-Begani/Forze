# Runbook — Apply migrations 033 → 045 to the live Supabase DB

**Why this exists:** the repo has no migration runner; migrations 033–044 shipped in code but were never applied to the live DB. Until they are applied, live users see: CRM **Pipeline & Deals tabs 500** (038 missing), **scheduled/drip/follow-up campaign sends silently do nothing** (033), landing-lead → campaign auto-enroll dead (041), Lead Scout weekly gate broken (042), public-endpoint rate limits no-op (044). Migration 045 (new) adds hot-path indexes and hardens `merge_venture_context`.

> **Fastest path (recommended):** open the Supabase dashboard → SQL editor, paste the **entire** `db/apply-pending-033-045.sql` file, and Run. That single file is all of 033–045 concatenated in order (idempotent). Then jump to the "Final all-clear" query at the bottom of this doc to confirm. The per-file steps below are the fallback if you want to apply/verify one at a time or the one-shot hits an error.
>
> **Note:** a `.md` file is not runnable SQL — it's this checklist. Run the `.sql` files (or the consolidated one above), not this document.

**How to run (per-file):** open the Supabase dashboard → SQL editor. For each step below, open the named file in `db/migrations/`, paste the **entire file**, run it, then run the verification query and confirm the expected result. Every file is additive + idempotent — **re-running any file is always safe**.

**Rules:**
- Apply **in numeric order**, one file per SQL-editor execution (do NOT concatenate files — 033 adds an enum value, and Postgres forbids using a new enum value in the same transaction that added it; separate executions are categorically safe).
- **034 and 040 must both be applied in the same sitting.** 040 fixes an RLS recursion bug in the policies 024/034 rely on — never apply 034 and stop.
- If any run errors with "unsafe use of new value", just **run that same file a second time** — the guards skip the already-added value and the rest executes.
- Deploy order vs code: the app code is fail-open/fallback for every one of these — **code-first or SQL-first are both safe**.

## Steps

### 1. `033_outreach_engine.sql` — scheduled/drip/follow-up sends start working
```sql
select 1 as scheduled_enum_ok from pg_enum e join pg_type t on t.oid = e.enumtypid
where t.typname = 'campaign_status' and e.enumlabel = 'scheduled';
```
Expect: one row.

### 2. `034_crm_rls.sql` — RLS on leads/analytics/outreach/testimonials (⚠️ continue to 040 in the same sitting)
```sql
select count(*) as crm_policies from pg_policies
where tablename in ('leads','analytics_events','outreach_campaigns','outreach_messages','testimonials','platform_feedback');
```
Expect: ≥ 6.

### 3. `035_crm_leads_unify_social.sql` — social leads persisted into `leads`
```sql
select to_regclass('public.idx_leads_venture_external_identity') as ok;
```
Expect: non-null.

### 4. `036_crm_lead_fields.sql` — company/phone/tags/owner on leads
```sql
select count(*) as new_lead_columns from information_schema.columns
where table_name = 'leads' and column_name in ('company','phone','tags','owner_id');
```
Expect: 4.

### 5. `037_crm_lead_activity.sql` — notes + activity timeline
```sql
select to_regclass('public.lead_activity') as ok;
```
Expect: non-null.

### 6. `038_crm_deals_and_stages.sql` — **fixes the 500ing Pipeline & Deals tabs**
```sql
select to_regclass('public.deals') as deals, to_regclass('public.pipeline_stages') as stages;
```
Expect: both non-null.

### 7. `039_crm_outreach_replies.sql` — persisted CRM replies
```sql
select to_regclass('public.outreach_replies') as ok;
```
Expect: non-null.

### 8. `040_fix_venture_members_rls_recursion.sql` — REQUIRED companion to 034
```sql
select proname from pg_proc where proname = 'is_venture_member';
```
Expect: one row.

### 9. `041_lead_campaign_bridge.sql` — landing leads auto-enroll into campaigns
```sql
select count(*) as bridge_columns from information_schema.columns
where (table_name = 'campaign_leads' and column_name = 'lead_id')
   or (table_name = 'campaigns' and column_name = 'auto_enroll_landing_leads');
```
Expect: 2.

### 10. `042_lead_scout_action.sql` — Lead Scout weekly metering
```sql
select pg_get_constraintdef(oid) as def from pg_constraint
where conname = 'feature_usage_counters_feature_id_check';
```
Expect: definition contains `lead_scout`.

### 11. `043_backfill_missing_venture_members.sql` — heal missing owner memberships
```sql
select count(*) as owners_missing_membership from ventures v
where not exists (
  select 1 from venture_members m where m.venture_id = v.id and m.user_id = v.user_id
);
```
Expect: 0.

### 12. `044_anon_rate_limit.sql` — public-endpoint rate limits become real
```sql
select to_regclass('public.anon_rate_limit_events') as tbl,
       (select proname from pg_proc where proname = 'record_anon_rate_limit_event') as fn;
```
Expect: both non-null.

### 13. `045_conversations_indexes.sql` — hot-path indexes + hardened context merge
```sql
select count(*) as conv_indexes from pg_indexes
where tablename = 'conversations' and indexname like 'idx_conversations_%';
```
Expect: 2. (Also adds `idx_ventures_project` and the COALESCE-hardened `merge_venture_context`.)

## Final all-clear (run once at the end)

```sql
select
  to_regclass('public.deals')                                        as deals,
  to_regclass('public.pipeline_stages')                              as pipeline_stages,
  to_regclass('public.lead_activity')                                as lead_activity,
  to_regclass('public.outreach_replies')                             as outreach_replies,
  to_regclass('public.anon_rate_limit_events')                       as anon_rate_limits,
  to_regclass('public.idx_conversations_venture_module_time')        as conv_index,
  (select 1 from pg_enum e join pg_type t on t.oid = e.enumtypid
    where t.typname = 'campaign_status' and e.enumlabel = 'scheduled') as scheduled_enum,
  (select 1 from pg_proc where proname = 'is_venture_member')         as rls_fn,
  (select 1 from pg_proc where proname = 'record_anon_rate_limit_event') as ratelimit_fn;
```
Expect: **every column non-null**.

## Post-apply smoke checks (in the app)

1. Open a venture → CRM → **Pipeline** and **Deals** tabs load without a 500.
2. Submit a test lead on a published landing page → lead appears in CRM; if a campaign has auto-enroll on, the lead appears as a pending recipient.
3. Wait for (or curl) `/api/cron/run-outreach` → response reports the tick without errors; a scheduled campaign becomes eligible.
4. Submit 6+ rapid test submissions to a public feedback form from one machine → later ones get rate-limited (429) — proof 044 is live.
